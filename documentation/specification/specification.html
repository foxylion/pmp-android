Einleitung

Zweck

Dieses Dokument stellt die Spezifikation des Projektes "Privacy Management Platform" dar. Es summiert die Ergebnisse der Analyse und repräsentiert die Grundlage für die spätere Entwicklung in einer Iteration. Es ist daher eine vertragliche Grundlage für Kunde und Entwickler, um die zu entstehende Software und ihre Funktionalität so ausführlich und vollständig wie möglich zu beschreiben, damit die weitere Entwicklung stets auf dieses Dokument zurückgreifen kann.

Die Spezifikation definiert dabei einen grafischen Prototypen für die spätere Benutzeroberfläche, beschreibt die gewünschten Funktionsabläufe und definiert im Begriffslexikon die Bedeutung von speziellen Begriffen, die in der Arbeitsumgebung verwendet werden. Da auch Begriffe neu definiert werden, um die Software möglichst knapp und für alle eindeutig zu beschreiben, empfiehlt es sich zunächst, einen Blick in das Begriffslexikon zu werfen und die Beschreibung vermeintlich klarer Begriffe dort nachzulesen.

Projektüberblick & Motivation

Die rasche Entwicklung in der Kommunikationsindustrie mit ihren unzähligen Möglichkeiten übersteigt die heutige Konfigurationssoftware. Nicht jeder kennt alle Anwendungen auf seinem Smartphone, ganz davon abgesehen welche Rechte die einzelnen Programme besitzen, ob diese auf den Internetanschluss zugreifen oder sogar Ihre E-Mails weiterleiten.

Im Rahmen des Studienprojekts "Privacy Management Platform" kurz "PMP" an der Universität Stuttgart soll eine Software entwickelt werden, die dem Konsumenten das sichere Benutzen eines Smartphones erlauben könnte.

Der Benutzer soll alle Apps und Ressourcen auf dem Smartphone schnell und einfach verwalten können. Mittels einer ansprechenden grafischen Bedienungsoberfläche wird es möglich sein jeder einzelnen Anwendung, die mit PMP kompatibel ist, bestimmte Rechte zuzuweisen bzw. diese zu entziehen.

Leserkreis

Diese Spezifikation dient als ein zentrales Dokument für die folgenden Lesergruppen:

  • den Entwicklern von PMP
  • dem Kunden
  • dem Betreuern des Studienprojekts

Konventionen

In diesem Dokument werden mehrere Konventionen verwendet, um knapp, übersichtlich, lesbar und treffend zu sein:

  • Für Überbegriffe wie Benutzer oder Entwickler wird das generische Maskulinum verwendet. Da die Deutsche Sprache hier keine zufriedenstellende Alternative kennt, sind damit selbstverständlich auch alle weiblichen Leserinnen angesprochen.
  • In diesem Dokument wird die deutsche Sprache verwendet; dabei ist allerdings zu beachten, dass sich übernommene oder übersetzte Begriffe aus dem Englischen nicht vermeiden lassen. Diese werden dann jedoch grammatikalisch an die Deutsche Sprache angepasst.
  • Ein Verweis in diesem Dokument auf andere Kapitel wird so dargestellt:
  • Die Versionsnummern sind nach dem Schema aufgebaut, dass zuerst die Iterationsnummer genannt wird und anschließend die Versionsnummer der Spezifikation in dieser Iteration. "3.2" bezeichnet z.B. die 2. Version der Spezifikation für die 3. Iteration.

Name Ressourcen anzeigen

Service Level einer App anzeigen bzw. ändern

Use-Case ID #3
Name Service Level einer App anzeigen bzw. ändern
Ziel Der Benutzer möchte das Service-Level einer App anzeigen bzw. ändern
Vorbedingungen
  • Der Benutzer hat die PMP-App gestartet und ist noch in keinem Untermenü.
  • Es ist mindestens eine App installiert, die PMP verwaltet.
Nachbedingungen Der Benutzer hat das Service-Level einer App angezeigt bzw. geändert.
Nachbedingungen im Sonderfall 1 Der Benutzer hat das Service-Level einer App festgelegt bzw. geändert.
Nachbedingungen im Sonderfall 2 Der Benutzer hat das Service-Level einer App nicht festgelegt bzw. nicht geändert.
Akteure Benutzer
Normalablauf
  1. Der Benutzer wählt die Übersicht über die installierten Apps aus, die PMP verwaltet.
  2. PMP listet dem Benutzer alle Apps auf.
  3. Der Benutzer wählt eine App aus.
  4. Der Benutzer wählt für diese App aus, dass er die Service-Level Übersicht anzeigen will.
  5. PMP listet dem Benutzer die Service-Levels der App als Übersicht auf und falls ein Service-Level festgelegt ist, wird dieser hervorgehoben.
  6. Der Benutzer schließt die Service-Level übersicht.
  7. Der Benutzer schließt die Übersicht der Apps.
Sonderfall 15a
  1. Der Benutzer wählt ein Service-Level aus.
  2. PMP stellt die Details zu diesem Service-Level dar.
  3. Der Benutzer bestätigt die Auswahl dieses Service-Levels.
  4. zurück zu Normalablauf Schritt 6
Sonderfall 25b
  1. Der Benutzer wählt ein Service-Level aus.
  2. PMP stellt die Details zu diesem Service-Level dar.
  3. Der Benutzer bestätigt die Auswahl dieses Service-Levels nicht.
  4. zurück zu Normalablauf Schritt 6

Anforderungen an PMP

Oberfläche

Die Oberfläche von PMP soll sich nahtlos in die Oberfläche von Android einfügen. Dazu gehört eine konsistente und logische Anordnung der Bedienelemente genauso wie nachvollziehbare und möglichst einfach Menüstruktur. Es gelten die Android User Interface Guidelines.

Import/Export

Keine Import-/Exportfunktionen in M-1 geplant.

Standardeinstellungen

PMP soll ohne weitere Konfiguration sofort nach der Installation ("Out of the Box") einsatzbereit sein.

Nicht-Funktionale Anforderungen

Bedienbarkeit

PMP-Android soll einfach bedienbar sein. Im einfachen Modus muss jeder Nutzer ohne weitere Hilfestellungen oder Einarbeitungszeit die grundlegenden Einstellungen ändern können und das Programm so intuitiv bedienen können, dass PMP nur minimal in seinen gewohnten Workflow eingreift. Es ist kein Expertenmodus für M-1 geplant.

Er hat jedoch gehört dass PMP ihm dabei helfen kann Strom zu sparen und seine Internetkosten zu senken indem er bestimmten Programmen die Nutzung energieintensiver Features oder die Nutzung des Internets verbietet. Von Sicherheitskonzepten, Nutzerrollen oder Ressourcen hat er jedoch keine Ahnung.

Quantitative Anforderungen

Leistungsanforderungen

Die Software soll basierend auf der Android Version 2.1.1 laufen. Alle weiteren Versionen aufwärts (2.2, 2.3, 2.3.3) sollen ebenfalls unterstützt werden.

Mengengerüst

Die Software kann ohne weitere Probleme mind. 30 Apps und mind. 20 Ressourcen verwalten. Die maximale Anzahl kann je nach Gerät variieren.

In der 1. Iteration werden fünf Standard-Ressourcen bereitgestellt. Nach jeder Iteration werden nach und nach weitere Standard-Ressourcen verfügbar sein.

Eingabelängen

Die PMP muss von einfachen Eingaben, wie „erlauben“ und „nicht erlauben“, bis zu komplexeren Eingaben, wie Datum, Uhrzeit und Kilometer verarbeiten können.

Je nach Einstellung eines Service Levels, können Flags (Privacy Levels) gesetzt werden. Flags können verschieden Typen enthalten:

  • Binäre Eingaben, zum Beispiel „An/Aus“, „Ja/Nein“
  • Datum
  • Wochentage
  • Uhrzeit
  • Koordinaten
  • Geschwindigkeit
  • Kilometer
  • …

Robustheit

Durch Verwalten mehrerer Ressourcen zur gleichen Zeit, besteht ein gewisses Absturzrisiko. Daher soll die PMP robust entwickelt werden.

Bei fehlerhaften Eingaben soll der Benutzer entsprechende Meldungen erhalten, wie zum Beispiel bei der Einstellung der Flags bezüglich falscher Eingaben der Zeit oder des Datums. (Usability Patterns).

Verfügbarkeit

Die PMP läuft während der Laufzeit des Geräts im Hintergrund, um die Apps und Ressourcen zu verwalten.

Falls PMP unerwartet beendet werden sollte, soll sie automatisch neugestartet werden.

Sicherheit

Bei wichtigen Einstellungsänderungen wird der Benutzer einmalig pro Sitzung nach einem Passwort gefragt. Während der Passworteingabe kann das Passwort angezeigt werden.

Portabilität & Kompatibilität

Die App soll auf allen Android–fähigen Smartphones ab der Version 2.1.1 kompatibel sein. Die PMP besitzt keine Einschränkungen bezüglich der Auflösung.

Internationalisierung

Die Oberfläche von PMP wird zunächst auf Englisch entwickelt werden, soll aber für beliebig viele Sprachen erweiterbar sein. Das Handbuch wird auf Deutsch entstehen, wobei die Screenshots die englische Oberfläche zeigen werden.

Protokollierung

Installation

Im Handbuch wird eine genaue Beschreibung der Installation vorliegen.

Erweiterbarkeit

Die Software soll hinsichtlich folgender Punkte erweiterbar sein:

  • Internationalisierung (bisher englische Benutzeroberfläche)
  • Ressourcen/Ressourcengruppen
  • Rollen

Wartbarkeit

Die Software soll hinsichtlich neuer Android-Versionen angepasst werden.

Abgeschlossenheit

Gesetzliche Einschränkungen

Es gibt keine generellen gesetzlichen Richtlinien oder Normen, die eingehalten werden müssen. Die Universität Stuttgart muss ein zeitlich und personell unbegrenztes Nutzungsrecht an der Software erhalten. Die Lizenzierung der Software an sich ist nicht vorgeschrieben, allerdings muss eine Open-Source-Lizenz verwendet werden.

App

BegriffApp
Synonyme-
BedeutungEine App ist eine lauffähige Software, die auf einem Mobilgerät ausgeführt wird. Eine App definiert verschiedene Service Level, die jeweils mehrere Services enthalten können. Eine App kann über eine Schnittstelle Ressourcengruppen über PMP benutzen. Eine App kann eigene Ressourcengruppen und Rollen in die PMP einfügen.

BezeichnungEine App wird durch ihren Namen identifiziert und durch ihre Funktionalität beschrieben.

Synonyme-

Privacy Level

BegriffPrivacy Level
SynonymeFlag
BedeutungEin Privacy Level ist eine Ressourcen-Einstellung in PMP, die durch die Ressourcengruppe definiert wird. Ein Privacy Level kann kontext-abhängig sein. Ihre konkrete Konfiguration für eine App wird beim Zugriff auf eine Ressource der Ressourcengruppe mitübergeben. Damit kann die Ressource über die Privatssphäre der zur Verfügung gestellten Inhalte entscheiden.
BeispieleDie App "Schachspiel" könnte Privacy Flags wie "Lesen" oder "Schreiben" anbieten.
AbgrenzungEin Privacy Level ist weder ein Service Level noch eine Rolle.
GültigkeitDer Privacy Level bleibt gültig, solange die entsprechende Ressourcengruppe in PMP eingebunden ist.
BezeichnungDer Privacy Level wird durch die zugehörige Ressourcengruppe und seinen Namen eindeutig identifiziert.

Rolle

BegriffRolle
SynonymePreset
BedeutungEine Rolle ist eine Zusammenfassung der konkreten Konfiguration einer Menge von Privacy Levels. Eine Rolle kann kontext-abhängig sein. Eine Rolle kann einer App zugewiesen werden.
BeispieleEine Rolle könnte eine Zusammenfassung aller Lese-Rechte auf alle Datenbanken sein.
AbgrenzungEine Rolle ist keine Zusammenfassung einer Menge von Privacy Levels. Eine Rolle ist keine feststehende Bezeichnung für eine App.
GültigkeitDie Rolle bleibt gültig, solange die entsprechende App in PMP eingebunden ist. Eine Rolle kann nur entstehen, wenn die enthaltenden Privacy Level bereits bekannt sind.
BezeichnungDie Rolle wird durch die zugehörige App und ihren Namen eindeutig identifizert.
